# Fix Sentry 4.5K Error Flood

## Context

The Sentry dashboard at `foolish.sentry.io` shows **4.5K errors** and **2 issues** in the last 24 hours with continuous spikes. Root cause investigation reveals:

1. **Codex CLI auth is broken**: `~/.codex/auth.json` contains `{"auth_mode":"chatgpt","tokens":{}}` — empty tokens. Every `codex exec` call fails with `missing field 'id_token'`.
2. **13 of 14 running agents use `provider=codex`** — so every LLM request from those agents fails immediately.
3. **Every failure generates a Sentry error event** via `sentry::capture_message` in `agent_loop.rs:1120` with no rate limiting.
4. **`traces_sample_rate = 1.0`** (100% sampling) in production amplifies event volume.
5. **Config corruption**: `~/.openfang/config.toml` has corrupted `sentry.tags` — `oyment = "local"` (truncated key) and duplicate `team = "foolish"`.

## Fixes

### 1. Fix config.toml corruption and reduce sampling
**File**: `~/.openfang/config.toml`
- Remove corrupted `oyment = "local"` tag
- Remove duplicate `team = "foolish"` line
- Reduce `traces_sample_rate` from `1.0` to `0.2` (20%) — still good observability, 5x fewer trace events

### 2. Add Sentry error rate limiting in agent_loop.rs
**File**: `crates/openfang-runtime/src/agent_loop.rs`
- Add a per-error-category rate limiter using a simple `DashMap<String, Instant>` with a 60-second cooldown per category
- Wrap the `sentry::capture_message` call at line ~1120 (generic LLM errors) and line ~1018 (rate limit errors) with the rate limiter
- This prevents identical error categories from flooding Sentry — first occurrence sends immediately, subsequent ones within 60s are suppressed (only breadcrumbs are added)

### 3. Re-authenticate Codex CLI (user action)
- User must run `codex login` to refresh the ChatGPT OAuth tokens
- This is the actual fix for the error source — without valid tokens, all codex-provider agents will keep failing

### 4. Restart daemon after fixes
- Rebuild with `cargo build --release -p openfang-cli`
- Restart daemon to pick up config changes and new rate-limiting code

## Verification
1. `cargo build --workspace --lib` — compiles
2. `cargo test --workspace` — tests pass
3. `cargo clippy --workspace --all-targets -- -D warnings` — zero warnings
4. After `codex login` + daemon restart: Sentry error rate should drop to near-zero
5. Check Sentry dashboard after 15 minutes to confirm error volume taper
