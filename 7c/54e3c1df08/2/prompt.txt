# Plan: Subprocess LLM Drivers for Claude Code + Codex CLI

## Context

OpenFang agents need LLM access, but the user authenticates via **Claude Code OAuth** and **Codex CLI ChatGPT OAuth** — neither token works against standard API endpoints (`api.anthropic.com` rejects OAuth, `api.openai.com` returns "insufficient_quota" for ChatGPT tokens). The solution: spawn the CLI binaries as subprocesses, since both `claude -p` and `codex exec` handle auth internally.

**Verified working:**
- `claude -p "prompt" --output-format json` → returns JSON with `result`, `usage`, `total_cost_usd`
- `codex exec "prompt" --json` → returns JSONL with `item.completed` events and `turn.completed` usage

## Changes

### 1. New file: `crates/openfang-runtime/src/drivers/claude_code.rs`

**ClaudeCodeDriver** struct (no fields needed — stateless subprocess spawner):

```rust
pub struct ClaudeCodeDriver;
```

**`complete()` implementation:**
1. Convert `CompletionRequest` messages → single prompt string (role-prefixed lines)
2. If `request.system` is set, pass `--system-prompt`
3. Spawn: `claude -p --output-format json --model <request.model>`
4. Pipe prompt to stdin, read stdout
5. Parse JSON: extract `result` → `ContentBlock::Text`, `usage.input_tokens` + `usage.output_tokens` → `TokenUsage`
6. Timeout: 120 seconds
7. On error: map to `LlmError::Api` or `LlmError::Http`

**No tool use** — CLI doesn't support OpenFang's tool protocol. Return text-only responses.
**No custom stream impl** — use default (wraps `complete()`).

### 2. New file: `crates/openfang-runtime/src/drivers/codex_cli.rs`

**CodexCliDriver** struct (no fields):

```rust
pub struct CodexCliDriver;
```

**`complete()` implementation:**
1. Convert messages → single prompt string
2. Spawn: `codex exec --json --model <request.model> "<prompt>"`
3. Read stdout JSONL, collect all `item.completed` events where `item.type == "agent_message"`
4. Concatenate text from agent_message items
5. Parse `turn.completed` event for `usage.input_tokens` + `usage.output_tokens`
6. Timeout: 120 seconds
7. Codex has no `--system-prompt` flag — prepend system prompt to the user prompt if present

```python
import sentry_sdk

def run_agent(agent_id, agent_name, agent_type, model, provider, session_id, user_id):
    with sentry_sdk.start_transaction(op="agent.run", name=f"Agent: {agent_name}") as tx:
        # Set tags — these match your highlighted tags in Sentry
        sentry_sdk.set_tag("agent_id", agent_id)
        sentry_sdk.set_tag("agent_name", agent_name)
        sentry_sdk.set_tag("agent_type", agent_type)
        sentry_sdk.set_tag("model", model)
        sentry_sdk.set_tag("provider", provider)
        sentry_sdk.set_tag("session_id", session_id)

        # Set user context
        sentry_sdk.set_user({"id": user_id, "email": "user@example.com"})

        # Set agent context (maps to your "agent" highlighted context)
        with sentry_sdk.configure_scope() as scope:
            scope.set_context("agent", {
                "agent_name": agent_name,
                "session_id": session_id,
                "agent_id": agent_id,
            })

        try:
            # Wrap each tool call as a child span
            with sentry_sdk.start_span(op="agent.tool_call", description="tool_name_here") as span:
                sentry_sdk.set_tag("tool_name", "tool_name_here")
                sentry_sdk.set_tag("step_name", "step_1_retrieval")
                # ... your tool logic here

        except Exception as e:
            sentry_sdk.capture_exception(e)
            raise
```

Review your recent changes with three parallel expert agents, apply worthwhile fixes, return a summary.

## How It Works

1. **Capture the diff** — get the real `git diff` of staged + unstaged changes (or last commit if clean)
2. **Launch 3 parallel review agents** on the diff:

   - **Efficiency Agent** — flags performance/resource issues (dead code, missed concurrency, hot-path bloat)
3. **Triage findings** — filter to only worthwhile fixes (skip nitpicks)
4. **Apply fixes** — implement the accepted improvements
5. **Summary** — concise report of what changed and why

### Finding 5: `SentryTracingMode` fields are structurally redundant

**File:** `/Users/gaganarora/Desktop/my projects/open_fang/crates/openfang-cli/src/main.rs` lines 709-729

**The code:**
```rust
#[derive(Clone, Copy)]
struct SentryTracingMode {
    enabled: bool,
    emit_error_events: bool,
}
```

**The issue:** Looking at `load_sentry_tracing_mode` (line 715), `sentry_enabled` is computed as `dsn_configured && (error_tracking || performance_monitoring)`. Meanwhile, `emit_error_events` is set to `config.sentry.error_tracking` directly -- without gating on `enabled`. This means `emit_error_events` can be `true` even when `enabled` is `false`. The more significant issue is that this represents three possible states with two bools (4 possible combinations):

| `enabled` | `emit_error_events` | Meaning |
|---|---|---|
| false | * | Sentry off (error_events ignored) |
| true | false | Sentry on, breadcrumbs only |
| true | true | Sentry on, errors promoted to events |

This maps cleanly to an enum with 3 variants, which eliminates the impossible state.

**Concrete fix:**

```rust
#[derive(Clone, Copy)]
enum SentryTracingMode {
    Disabled,
    BreadcrumbsOnly,
    WithErrorEvents,
}

impl SentryTracingMode {
    fn is_enabled(self) -> bool {
        !matches!(self, Self::Disabled)
    }

    fn emit_error_events(self) -> bool {
        matches!(self, Self::WithErrorEvents)
    }
}
```

Then `load_sentry_tracing_mode` becomes:
```rust
fn load_sentry_tracing_mode(config_path: Option<&PathBuf>) -> SentryTracingMode {

    let dsn_configured = config.sentry.dsn.as_ref().is_some_and(|dsn| !dsn.trim().is_empty());

        return SentryTracingMode::Disabled;
    }
    if config.sentry.error_tracking {
        SentryTracingMode::WithErrorEvents
    } else {
        SentryTracingMode::BreadcrumbsOnly
    }
}
```

```rust
    }
    let emit = mode.emit_error_events();
```

### Finding 4: Subprocess timeout does not kill the child process

**Files and lines:**
- `/Users/gaganarora/Desktop/my projects/open_fang/crates/openfang-runtime/src/drivers/claude_code.rs` lines 64-67
- `/Users/gaganarora/Desktop/my projects/open_fang/crates/openfang-runtime/src/drivers/codex_cli.rs` lines 55-58

**The specific inefficiency:**

Both drivers use `tokio::time::timeout(120s, child.wait_with_output())`. When the timeout fires, the `Elapsed` error is returned and the function exits, but `child.kill()` is never called. The subprocess continues running as an orphan.

The code pattern is:
```rust
let output = tokio::time::timeout(
    std::time::Duration::from_secs(120),
    child.wait_with_output()
)
.await
.map_err(|_| LlmError::Http("...timed out...".to_string()))?
```

When the timeout fires, `child` is dropped. For `tokio::process::Child`, dropping does **not** kill the child process -- it detaches it. The tokio documentation explicitly states: "dropping a Child does not kill the child, but rather detaches the child from the current process and may cause it to keep running in the background." This means a hung `claude` or `codex` CLI process will continue consuming resources indefinitely.

**Estimated impact: MEDIUM.** Under normal operation, 120 seconds is generous and timeouts should be rare. But when they do occur (e.g., CLI waiting for interactive auth, network stall), each orphaned process consumes memory and potentially holds file locks. Repeated timeouts could accumulate orphaned processes.

**Concrete fix:**
Add explicit kill logic when the timeout fires:
```rust
let output = match tokio::time::timeout(Duration::from_secs(120), child.wait_with_output()).await {
    Ok(result) => result.map_err(|e| LlmError::Http(format!("...process error: {e}")))?,
    Err(_) => {
        // Kill the child to avoid orphaned processes
        let _ = child.kill().await;
        return Err(LlmError::Http("...timed out after 120s".to_string()));
    }
};
```

Note: Since `child.wait_with_output()` consumes `self`, the kill approach requires restructuring to use `child.kill()` before `wait_with_output()` consumes ownership. The idiomatic approach is to `tokio::pin!` a kill-on-drop wrapper, or to use `child.kill()` + `child.wait()` separately.
